## Phase 2 — Auth & tenancy

```
Implement invite-only, domain-restricted EMAIL/PASSWORD auth via Supabase Auth per
docs/01-ARCHITECTURE.md §4. This is NOT an OAuth provider / social login — do not wire one.
Follow the single API-driven membership flow exactly (no auth.users trigger).

First, define the API contract in packages/shared as Zod input/output schemas for every endpoint
below (docs/01 §9) — build this before the routes.

Backend (apps/api):
- supabaseAdmin client (service role, env only). Auth middleware: verify the Supabase JWT, attach
  { userId, orgId, role } DERIVED FROM THE JWT (never the request body); ownership-check path :ids.
- POST /invites (admin): validate email domain ∈ org.allowed_domains, create invites row, send the
  Supabase invite email. POST /invites/:id/revoke. GET /invites (admin).
- POST /invites/accept: re-validate the email domain (defense in depth), create the memberships row
  (user→org→role), write an audit entry. NO trigger on auth.users.
- Custom Access Token hook SQL: look up membership by user_id, inject org_id + role; if none, inject
  no org claim. Provide enable instructions.

Frontend (apps/web):
- Supabase client (anon key + persisted session).
- AuthLayout using TemplatesTailwind sign-in form: login, set-password / accept-invite pages.
- After /invites/accept, call supabase.auth.refreshSession() to obtain claim-bearing token.
- Add an explicit "Account pending / no organization" state for a logged-in user with no membership
  (never a blank RLS-denied screen).
- Pinia session store: user, org, role, membership; load on boot; refresh.
- Vue Router guards; role-aware AppShell (sidebar template): Driver sees Fuel Log + own history;
  Manager/Admin everything; Auditor read-only.
- Settings → Users screen (admin): send/revoke invites, change roles (table + slideover templates).

Enforce that a non-@silvicominc.com email cannot be invited OR accept. Confirm open signups are
disabled in Supabase. Add Vitest for domain-check + role-guard, and an RLS cross-tenant denial test
(docs/01 §10).
```

## Phase 4 — Fuel capture

```
Build fill-up capture per docs/00 §5 and the fuel_transactions table in docs/02 §3.
Mobile-first — a driver must complete it in well under 30 seconds on a phone.

- Generate the fill-up id CLIENT-SIDE (UUID v4) and use it as both the row PK and the storage path
  (idempotency + known path before upload — docs/02 §10.2, §10.4).
- Fill-up form (forms template, single column, large touch targets): vehicle picker (defaults to
  the driver's assigned vehicle), fueled_at (defaults now, real instant), odometer, gallons,
  total_cost (gallons + total are authoritative; price_per_gal is DERIVED = total/gallons, 3dp),
  location_text, and an OPTIONAL receipt photo client-compressed to ≤200 KB WebP, uploaded to
  bucket `receipts` at org_id/vehicle_id/{fillup_uuid}.webp.
- INLINE validation BEFORE submit:
    * odometer below the vehicle's last reading → red warning
    * missing odometer → amber nudge
    * gallons > tank_capacity_gal → HARD CONFIRM ("exceeds tank capacity, will be flagged —
      submit anyway?") to separate typos from genuine theft capture (docs/06 M10).
  Use a shared Zod schema; warnings computed from the vehicle's current data via Vue Query.
- On submit, write the transaction via Supabase client. (Engine scoring is wired in Phase 5.)
- Fuel Log list: KEYSET-paginated, filter by vehicle/driver/date, show computed_mpg once available.

Add Vitest for the price/total/gallons derivation and the warning/confirm logic.
```

## Phase 5 — Anomaly engine

```
Implement the anomaly engine exactly per docs/02-DATA-MODEL.md §7 and docs/01 §5.

Use the PRECISE definitions in docs/02 §10.7–10.8 — do not improvise: previousTxn = greatest
fueled_at strictly < current (tiebreak created_at, same vehicle); guards (skip MPG when gallons<=0
or no baseline or invalid delta; skip jump when hours_elapsed<=0; first fill skips delta rules);
fuel-type gating (Tier 2/3 only for diesel/gasoline); precedence (capacity suppresses top-off);
off-hours via org tz; cost_outlier fixed band; disabled_rules opt-out.

packages/shared/anomalyRules.ts:
- Pure functions, one per ruleId across Tier 1–4, each (ctx) => RuleResult
  { ruleId, fired, severity, message, evidence }. ctx = { txn, vehicle, previousTxn,
  recentTxns, thresholds, operatingHours }.
- runAllRules(ctx) aggregator. Baseline MPG = rolling median of last 5 valid fills (≥3 required,
  else skip MPG rules), excluding odometer-anomalous fills, fallback to vehicles.baseline_mpg.
- FULL Vitest coverage: a passing AND a failing fixture for EVERY rule, plus the guard cases.

apps/api:
- POST /transactions/:id/score (service role): derive org_id from JWT, ownership-check the txn,
  take a per-vehicle advisory lock, load context, compute miles_since_last + computed_mpg, run
  runAllRules. RECONCILE anomalies (do NOT blanket-delete): replace still-open rules-anomalies;
  mark investigating/resolved/dismissed ones as 'superseded' with a note. Update has_anomaly,
  max_severity, and the derived vehicles.current_odometer.
- RE-SCORE CASCADE: scoring the changed txn also re-scores the immediately following txn in
  fueled_at order for that vehicle.
- Trigger: call POST /score from the web app right after the Supabase insert/edit (chosen
  mechanism; document it). Imports (Phase 4.5) enqueue scoring in fueled_at order.

Backfill: a one-off endpoint to score all seeded transactions in fueled_at order so the demo lights up.
```

  internal ids), compute geo facts, check cache (input_hash), call Claude (Haiku first; escalate to

  persist ai_verifications, set ai_risk_level, boost notification if risk_level>=high. ANTHROPIC_API_KEY
  server-only. Selective triggers: only txns with an anomaly severity>=medium. Enforce monthly token
  budget + kill-switch. Failures are non-blocking.
- POST /anomalies/:id/ai-examine (manager, on-demand, Sonnet).
- UI: "AI Assessment" card on anomaly detail (risk badge, summary, location verdict, factors,
  recommended action — labeled AI-generated/for review); queue gains an AI-risk sort/filter; a
  "re-examine" button. Write ai.verification_run audit entries.
- Treat all transaction free-text (station names) as untrusted; delimit in the prompt; never let it

## Tips for driving Windsurf well

- Keep each prompt to **one phase**; let it finish, review the diff, run lint+test, then continue.
- If it drifts from `/docs`, reply: *"Re-read docs/02-DATA-MODEL.md §X and align — don't invent schema."*
- Ask it to **write the tests first** for the anomaly rules; they're the riskiest logic.
- When a UI task starts, point it at the exact template path (e.g. *"use
  TemplatesTailwind/application-ui-v4/vue/forms/..."*) so it reuses rather than rebuilds.
- Commit at the end of every phase with a conventional message; tag `phase-N-done`.
